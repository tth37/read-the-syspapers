## Thesis

Containers stopped being an isolation research problem years ago. The namespaces-and-cgroups debate is settled, the kernel surface is hardened well enough that the industry now multi-tenants mutually distrusting workloads on the same host, and every major cloud runs its serverless tier on `runc`-shaped primitives behind MicroVMs. What the systems community still calls "container research" in 2025 is almost entirely a single problem wearing different hats: starting the thing is too slow, and the fix every recent paper converges on is the same one — stop booting, start restoring.

## The setup

The textbook story about containers is an isolation story. Process, mount, network, pid, user, and ipc namespaces for what the container sees; cgroups for what it can consume; seccomp-bpf for what it can ask the kernel. That story was the whole debate for most of the 2010s. The question was whether Linux's monolithic kernel could safely multi-tenant without a hypervisor, and the answer, bluntly, is that it mostly could — with the caveat that anywhere the guarantee had to be stronger, a MicroVM crawled back underneath the container. AWS Firecracker and Kata Containers won that round.

But the operational story is different. Once containers started running real production FaaS — [AWS Lambda](https://aws.amazon.com/blogs/compute/under-the-hood-how-aws-lambda-snapstart-optimizes-function-startup-latency/), Azure Functions, Ant Group's serverless fabric — the metric that mattered stopped being attack surface. It became the number of milliseconds between a request arriving and the first byte of user code running. That is the cold start, and it has quietly become the only remaining container-research frontier worth publishing in a top venue.

## The evidence

### The research has moved on

Look at where 2025's container-adjacent papers actually spend their effort. [AFaaS (Fork in the Road)](/en/papers/osdi-2025/fork-in-the-road-reflections-and-optimizations-for-cold-start-latency-in-production), from Tsinghua and Ant Group at OSDI '25, is a post-mortem of Ant's production FaaS: more than 50% of its functions have a cold-start probability above 0.75, more than 35% always cold start, and useful work often finishes in 50–100 ms. A one-second cold start is not tail-latency noise there; it is the dominant cost. AFaaS gets production cold starts down to 6.97–14.55 ms at 24× concurrency. The paper frames itself as "reflections and optimizations" — a genre marker that the field has stopped asking how to make containers safer and started asking why the last few seconds will not go away.

[Dandelion](/en/papers/sosp-2025/unlocking-true-elasticity-for-the-cloud-native-era-with-dandelion), from ETH Zurich at SOSP '25, is the same story with a different accent: compute sandboxes created in ~100 μs on CHERI and ~220 μs on plain KVM, so per-request cold starts become plausible, and warm pools — Knative's way of hiding cold starts by committing 16× more memory than it actually uses — can be cut 96% on the Azure Functions trace. [MettEagle](/en/papers/osdi-2025/metteagle-costs-and-benefits-of-implementing-containers-on-microkernels), also OSDI '25, builds container-style compartments on L4Re and reports ~1 ms cold start for an empty compartment against ~70 ms for runC. Even that paper — whose nominal contribution is an isolation argument — leads its results with the startup number. The cold-start story has become the story of containers *regardless* of whether the paper thinks it is making a security argument.

### The mechanism is always the same: stop booting, start restoring

What is striking about the cold-start papers is how uniformly they answer the question. Almost none of them try to make booting faster in the traditional sense. They all try to avoid booting by serializing a running instance's state and deserializing it somewhere else.

AWS's production answer, [Lambda SnapStart](https://aws.amazon.com/blogs/compute/starting-up-faster-with-aws-lambda-snapstart/), snapshots a Firecracker MicroVM's memory and disk after initialization, chunks it, caches it in a multi-tier fabric, and demand-loads the working set on restore. The hard part, as [Marc Brooker has written](https://brooker.co.za/blog/2022/11/29/snapstart.html), is not the snapshot format; it is distributing the snapshot data across a planet-scale fleet so that a demand page fault does not turn into a slow S3 fetch. [Modal's gVisor-based equivalent](https://modal.com/blog/mem-snapshots) restores `import torch` in ~1 s instead of ~5 s using a FUSE-backed page file. Kubernetes, through `kubelet`-driven CRIU, graduated container checkpointing to [beta in v1.30](https://kubernetes.io/blog/2022/12/05/forensic-container-checkpointing-alpha/) and shipped GPU support through [CRIUgpu in CRIU 4.0](https://www.devzero.io/blog/gpu-container-checkpoint-restore). The platforms have converged on "container = serializable machine state."

The papers converge too. [WorksetEnclave](/en/papers/asplos-2026/worksetenclave-towards-optimizing-cold-starts-in-confidential-serverless-with-workset-based-enclave-restore) at ASPLOS '26 takes the same idea to SGX, where snapshotting is harder because enclave memory is encrypted and sealed: it encrypts inside the enclave, records which pages actually get touched during profiling (the "workset"), restores only those pages into a freshly built EDMM enclave, and demand-loads the rest with integrity checks. Startup drops from seconds to under 400–600 ms, and 74–95% of the enclave memory is never restored because it turns out never to be read on the hot path. That is exactly the chunked-retrieval insight behind SnapStart, applied to a new memory hierarchy. Same algorithm, new substrate.

[AFaaS](/en/papers/osdi-2025/fork-in-the-road-reflections-and-optimizations-for-cold-start-latency-in-production) generalizes the snapshot from a flat image to a tree: a level-0 OS seed, a level-1 language-runtime seed, a level-2 user-code seed, and copy-on-write forks from the closest ancestor. That is a snapshot hierarchy. [BlitzScale](/en/papers/osdi-2025/blitzscale-fast-and-live-large-model-autoscaling-with-o-1-host-caching) at OSDI '25 applies the same reflex to LLM weights: one host-cached copy of the model anywhere in the cluster, multicast over RDMA to new instances, with partially loaded instances executing the layers they already have. The authors frame it as parameter broadcast, but the mechanism is snapshot-restore at the weight-tensor granularity — and the payoff is, again, a cold-start win measured in TTFT.

The filesystem papers tell the same story from the other side. [CoFS](/en/papers/fast-2026/cofs-a-filesystem-for-fast-container-startup) at FAST '26 observes that image pulling accounts for 76% of container startup but only 6.4% of the pulled bytes are ever read, builds a minimal-perfect-hash index offline into the image itself, and serves both metadata and cached reads from inside the kernel. That is snapshotting the image lookup table. [ThinkAhead](/en/papers/fast-2026/how-soon-is-now-preloading-images-for-virtual-disks-with-thinkahead) at the same venue learns per-image boot traces and preloads virtual-disk blocks before the first access; that is SnapStart's chunk-prediction story applied to EBS. The industry has already gone this way: [eStargz](https://github.com/containerd/stargz-snapshotter) and [zstd:chunked](https://github.com/containers/storage/pull/775) standardize lazy pulling so an image can start running before it finishes downloading, and [AWS's SOCI](https://engineering.grab.com/docker-lazy-loading) ships an OCI-compatible index that does the same thing without re-converting the image. Every one of these mechanisms is a snapshot of something — the filesystem tree, the working set, the page table, the weight tensor — plus a policy for streaming it in lazily.

### Even the networking papers are caching papers

It is tempting to fence off container networking as the one cold-start-free bastion of container research, but the framing has crept in there too. [OnCache](/en/papers/nsdi-2025/oncache-a-cache-based-low-overhead-container-overlay-network) at NSDI '25 observes that the overhead of a Kubernetes overlay is spread across veth traversal, conntrack, filters, routing, and VXLAN encapsulation — but once a flow is established, the *combined* cross-layer result is stable. So OnCache memoizes that result in eBPF maps and skips the overlay on the fast path. In the paper's own framing this is a caching optimization; structurally it is a snapshot of the networking fast path, restored per packet. [SG-IOV](/en/papers/asplos-2026/sg-iov-socket-granular-i-o-virtualization-for-smartnic-based-container-networks) at ASPLOS '26 makes a related move at the SmartNIC boundary: offload below-socket and above-socket functions onto the NIC together, so the "container network" becomes a set of pre-configured streams rather than a packet pipeline built fresh for each flow. Pre-configured means snapshotted.

The generalization is uncomfortable but concrete: the recent container systems literature is really one paper with many titles. It says that the expensive work in running a container is work that does not need to be redone, and the frontier is finding which slice of work — image metadata, page tables, enclave pages, weight tensors, flow-table entries, network fast paths — is the slice you can serialize once and paste in many times.

## The counter-evidence

The strongest objection to this reading comes from [Dandelion](/en/papers/sosp-2025/unlocking-true-elasticity-for-the-cloud-native-era-with-dandelion). Dandelion's argument is that snapshotting treats the symptom: the reason cold starts are expensive is that each function boots a POSIX-like sandbox with a guest OS and a networking stack, and the right fix is to change the programming model so the thing you start no longer has those components. If compute functions are pure and communication is pulled into trusted functions, sandboxes drop below 300 μs *without* snapshot machinery, and per-request cold starts become the default rather than the exception. Dandelion would say the snapshot-restore papers are attacking the wrong abstraction and should be replaced, not optimized. That is a genuine competing framing — if enough workloads migrate to a Dandelion-shaped decomposition, the cold-start-as-snapshot thesis becomes a historical artifact.

[Quilt](/en/papers/sosp-2025/quilt-resource-aware-merging-of-serverless-workflows) at SOSP '25 makes an adjacent move: instead of snapshotting the sandbox, fuse workflow functions that talk to each other into a single compiled binary, so the sandbox boundary the snapshot would have protected no longer exists. Median workflow latency drops 45–70% and throughput goes up 2–13×, mostly by removing inter-function RPC hops that cold-start penalties would otherwise amortize. The container is made cheaper by being made rarer. [MettEagle](/en/papers/osdi-2025/metteagle-costs-and-benefits-of-implementing-containers-on-microkernels) is a third kind of pushback: perhaps the cold-start cost comes from Linux's hardening machinery and the right fix is a different kernel, not a better snapshot. And [Leopard](/en/papers/nsdi-2025/making-serverless-pay-for-use-a-reality-with-leopard) at NSDI '25 argues that a real share of serverless inefficiency lives in the billing contract, not the sandbox mechanism.

Two caveats belong in the honest version of the thesis. First, isolation research is not dead; it has moved one level down the stack, into confidential computing (TEEs, attested enclaves, confidential VMs). That work is important, but it uses the container as a packaging unit rather than studying the container itself. Second, the snapshot-centric view is noticeably weaker for stateful workloads — OLTP, gaming sessions, long-lived connections — where "container state" is not a clean prefix of execution you can materialize and replay. The thesis is strongest where snapshots are strongest: short-lived, stateless, invoked often.

## What this means

If the thesis holds, it changes a couple of things about how to think about container systems in 2026.

For platform builders, the image is more important than the runtime. The security-isolation layer is commoditized; the piece that distinguishes fast platforms from slow ones is how well they can serialize, distribute, and demand-page state across a fleet. AWS shipped a multi-tier cache in front of [SnapStart](https://aws.amazon.com/blogs/compute/starting-up-faster-with-aws-lambda-snapstart/) before it shipped meaningful isolation improvements this decade. That is the right order of priorities if the thesis is right.

For researchers, the interesting primitive is the snapshot, not the namespace. A disproportionate amount of recent mechanism work — working-set tracking, content-addressed sharing, MPHF metadata indexes, multicast parameter distribution, memoized overlay paths — is an exploration of how to snapshot some specific piece of a running system. There is still enormous headroom in snapshotting the things we do not yet snapshot well: GPU kernel state (CRIUgpu is new), database buffer pools, JIT caches, connection-pool state.

For users, the corollary is that per-request sandboxing has become genuinely plausible. That was a fantasy when cold starts were seconds; it is an engineering choice at hundreds of microseconds. Once a sandbox costs no more to create than a function call, the ergonomic argument for warm pools — the thing keeping providers from offering true elasticity — collapses, and the security argument for running untrusted code in its own sandbox gets real.

The "container" label survives because the ecosystem around OCI images, registries, and Kubernetes is too large to rename. But the thing being researched under that label has quietly changed. Every new paper that makes containers faster is a snapshot paper in a trenchcoat — and the papers that are not are the ones arguing that snapshots are the wrong answer.
