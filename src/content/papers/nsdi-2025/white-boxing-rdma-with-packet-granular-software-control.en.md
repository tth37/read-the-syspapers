## TL;DR

SCR "white-boxes" RDMA by moving transport control, but not packet movement, into software on Nvidia BlueField-3's Datapath Accelerators. Its dequeue-rate control model plus multi-domain event collection let software implement fair scheduling, RTT-based congestion control, multipath routing, and receiver-driven backpressure while leaving the ASIC datapath and host software stack unchanged.

## Problem

RDMA gets its performance from ASIC-offloaded transport, but that also makes policy rigid. Shipping a new congestion controller, fairness mechanism, or routing policy usually means waiting for a vendor ASIC, building a custom NIC or FPGA, or inserting a host-side overlay above verbs. The first two are slow and expensive; the third only sees message-granular operations and still cannot directly steer the RNIC.

That mismatch worsens as RDMA moves into multi-tenant Ethernet for storage and GPU traffic. These deployments need controls beyond classic fabric congestion response: fairness across tenants with different message sizes, receiver-side feedback for incast, and path selection across multiple routes. Useful signals now arrive at packet or sub-microsecond granularity, while commodity SmartNIC cores are weak.

## Key Insight

The central move is to control the RNIC's per-QP dequeue rate: the rate at which the NIC pulls work queue elements and payloads from host memory. That knob is more general than a congestion-control sending rate because it simultaneously shapes wire bandwidth, PCIe DMA pressure, RNIC internal processing pressure, and indirectly the load seen by the peer receiver.

Once dequeue rate is the control variable, "white-boxing RDMA" becomes plausible. Hardware keeps the fast path for DMA, packetization, and transport execution, while software computes dequeue rates from diverse events. The RNIC remains a high-performance datapath, but its control plane becomes programmable in the spirit of a white-box switch.

## Design

SCR is organized around an event-driven framework. Event collectors pull signals from three domains: the fabric domain (`TX`, `ACK`, `NACK`, `CNP`, and RTT), the host domain (application hints, PCIe utilization or latency, and RNIC utilization), and the peer domain (receiver credits plus peer PCIe and RNIC pressure). When the RNIC cannot expose a signal directly, SCR uses lightweight out-of-band UDP packets generated by the DPA.

Keeping up with line rate is the hard part. BF3 hardware can already coalesce some in-band fabric events, but only with fixed policies, so SCR adds software queues and multiplexers with richer policies such as accumulation and keep-latest. Event processors use deterministic multithreading: run-to-completion handlers, shared-nothing state, and steering that maps the same flow and event type to the same DPA thread. The paper's "granularity invariance" argument says that fixed total line rate means more flows on a fixed thread pool need not reduce fidelity.

Because DPA cores are weak and caches are tiny, the implementation minimizes both state and instruction count. It uses Bloom filters, compact hash maps, fixed-point arithmetic, and a tiny host agent only for BF3 limitations such as pre-launching threads and relaying updates to `doca_pcc`.

## Evaluation

The prototype is 7.8K lines of C/C++ on two 100 GbE servers with BlueField-3 DPUs and A30 GPUs. With up to 64 DPA threads, SCR keeps up with line-rate traffic and maintains essentially invariant coalescing granularity even as the number of QPs scales to 1024. Software coalescing improves event rate by 51.3% over hardware-only coalescing in a 4-thread setting, and a Bloom filter shrinks one membership structure from about 2 MB to 8 KB for 8192 flows.

The strongest end-to-end result is the fair QP scheduler. For two competing QPs, Water Filling improves Jain's fairness index by up to 1.78x over the default scheduler; at 4 QPs, fairness rises from 0.51 to 0.96 while staying near line rate. In an isolation test with a 256 B latency-sensitive flow competing against a 64 KB bandwidth-hungry flow, Water Filling cuts latency from 5.3 us to 2.5 us, a 52.8% reduction, while keeping the large flow at 82.4 Gbps instead of the default 90.6 Gbps. Appendix cases show the same substrate can also realize Swift-like RTT control, multipath failure detection, 16-to-1 incast receiver credits, and fair sharing for GPU-Direct RDMA and NVMe-oF without host changes.

## Novelty & Impact

The novelty is not one new transport algorithm but a reusable control substrate for commercial RDMA NICs: a general dequeue-rate model plus a multi-domain event framework. It sits between coarse host overlays such as `Flor` and `Justitia` and bespoke NIC or FPGA designs, making it a plausible template for future RNIC APIs, SmartNIC firmware, or Falcon-style rate engines.

## Limitations

The prototype is tightly bound to BF3's exposed interfaces. BF3 lacks direct `RX` events, richer NIC counters, dynamic thread creation, and general DPA IPC, so SCR still needs a small host agent and awkward coordination with `doca_pcc`; some actions, such as per-packet header rewrites, require extra NIC flow-engine tricks.

Timing precision is also limited. The paper measures several microseconds between event generation and installed rate changes, and precise rate limiting below 1 Gbps remains weak. So the paper proves a flexible substrate, not that every fine-grained RDMA policy is immediately production-ready.

## Related Work

- _Li et al. (OSDI '23)_ - `Flor` exposes a software RDMA framework above heterogeneous RNICs, whereas `SCR` moves control into the packet-granular transport loop while preserving host transparency.
- _Montazeri et al. (SIGCOMM '18)_ - `Homa` motivates receiver-driven transport control, and `SCR` adapts that style of feedback to one-sided RDMA through out-of-band peer signals.
