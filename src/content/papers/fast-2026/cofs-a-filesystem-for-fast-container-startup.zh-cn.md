## TL;DR

`CoFS` 是一个基于扩展 `FUSE` 的容器镜像文件系统，它利用“镜像一经发布就对容器呈现只读且固定”的事实，在构建阶段预先生成最小完美哈希函数（`MPHF`），把大多数路径查找变成内核内的索引访问；同时它用稀疏 mirror file 保存已下载数据，让缓存命中读请求也能直接走内核态。这样一来，CoFS 既保留了按需拉取的灵活性，又显著削弱了 `FUSE` 用户态中转带来的冷启动开销。

## 问题背景

这篇论文关注的是容器在编排系统和类 serverless 场景中的冷启动。容器启动通常要串行经历镜像下载、镜像解包、容器配置和进程启动。作者引用前人工作指出，镜像拉取占到了容器启动时间的 `76%`，但真正会在启动阶段被读到的数据只占已下载数据的 `6.4%`。这说明“把整张镜像先完整拉下来”本身就和启动阶段的实际需求严重失配。

现有 lazy pulling 系统虽然减少了无效下载，但没有真正消除关键路径上的元数据和数据访问开销。`Nydus-fuse` 与 `eStargz` 这类文件系统级方案依赖用户态 daemon 来处理查找和读取，Linux 内核在做路径解析时会不断发起 `LOOKUP`，每一层目录都可能触发上下文切换和请求拷贝。`Nydus-erofs` 虽然把更多逻辑搬进了内核，但首次访问仍要经过 `fscache` 与用户态 backend，调用链更长，还伴随同步写入和缓存管理复杂性。真正的问题不是“少下载一些字节”这么简单，而是要把首次元数据访问和首次数据访问的固定开销压到足够低，lazy pulling 才会实实在在地缩短容器启动时间。

## 核心洞察

论文最核心的观察是：容器镜像层在构建后对容器而言是静态只读的，因此完全可以把一部分工作前移到离线构建阶段。CoFS 不再在运行时做通用目录查找，而是在构建镜像时为整棵文件树计算 `MPHF`，并把文件元数据按哈希值放进稠密数组里。由于镜像不会变，这个哈希布局可以做到无冲突且空间开销可控，于是运行时查找就退化为一次可预测的内核索引操作，而不是标准 `FUSE` 那种逐层向用户态询问。

相同的只读假设也适用于数据路径。CoFS 并不要求每次缓存命中的读请求都重新穿过用户态 daemon，而是把已下载的数据范围写入宿主机文件系统上的稀疏 mirror file。后续访问时，内核先判断请求区间是否已经物化；如果数据已经在本地，就直接通过宿主机文件系统读取。换句话说，CoFS 把问题拆成了两半：元数据用离线预计算索引解决，数据用细粒度稀疏缓存解决，从而把最常见的热路径尽可能留在内核态。

## 设计

CoFS 不是完全推倒重来，而是在 `eStargz` 的基础上定义了自己的镜像布局。它把原本放在 `stargz.index.json` 中、用于构造 inode 的那部分元数据抽出来，重组成一个二进制文件 `cofs.inode.array`。这个文件前部保存 `MPHF` 的参数 `{m, n, T1, T2, g}`，后面是稠密的 metadata array，再后面是额外元数据区，用于存放超长文件名和扩展属性。每条 metadata entry 大小固定为 `120` 字节；短文件名直接放在 entry 里，长文件名通过偏移量指向尾部区域。硬链接则通过分配多个 entry、重复记录相同 inode 元数据来维持“一把 key 对应一个槽位”的不变量。

第一组 `MPHF` 的 key 是 `(parent inode, filename)` 的串联值。运行时，`cofs-driver` 通过 `ioctl` 把 `MPHF` 参数读入内核，收到查找请求后计算哈希值，直接定位到 metadata array 中对应位置，再校验 parent inode 与文件名是否合法。若文件名较短，比较操作完全在固定长度 entry 内完成；若文件名超过 `16` 字节，则根据 entry 里记录的偏移去尾部元数据区取出完整名称。论文的重点在于，这条路径规避了普通 `FUSE` 在路径解析过程中反复进入用户态的成本。

数据路径由两个组件配合完成。`cofs-snapshotter` 是基于 `eStargz` snapshotter 改出来的 `containerd` snapshotter。容器创建前，它会异步拉取 `cofs.inode.array`，而不是整层镜像，并把该文件描述符通知给内核侧驱动；同时，它为每个 layer 准备一个 FUSE 挂载目录和一个 mirror 目录。真正发生 I/O 时，若某段数据本地还没有，用户态 daemon 就从远端 registry 拉取对应字节范围，并把数据异步写到以 inode 号命名的稀疏 mirror file 中。

`cofs-driver` 则在内核中把这些稀疏文件变成快速路径。处理读请求时，它先检查 mirror file 是否存在，再用 `vfs_lseek(..., SEEK_HOLE)` 判断目标区间是否已经下载完成；如果整个文件已经完整落地，或请求区间确定已存在，就直接调用 `vfs_read` 从宿主机文件系统读取，否则才像普通 `FUSE` 那样把请求转发给用户态 snapshotter。除此之外，CoFS 还构造了第二组以完整路径为 key 的 `MPHF`，并用 `kprobe` 挂到 `do_filp_open` 上，在路径深度大于三层时，从底向上并行预取深层 inode，以进一步加速路径解析。

## 实验评估

实验平台是 Linux `6.9.1`，双路 `Xeon E5-2640 v4`、`128 GB` 内存、`4 TB HDD`，并用一台通过 `1 GbE` 连接的远端机器充当镜像仓库，以模拟带宽受限的共享 registry。对比系统包括 `CoFS`、`CoFS-gzip`、`traditional`、`Nydus-fuse`、`Nydus-erofs` 和 `eStargz`，工作负载则是 `mariadb`、`redis`、`tomcat` 和 `elasticsearch` 四个常见服务镜像。

首先，CoFS 的镜像格式开销很小。以 `mariadb-10.7.3` 为例，`eStargz` 格式下镜像大小是 `126.2 MB`，CoFS 格式是 `128.4 MB`；构建时间则从 `23 s` 增加到 `25.36 s`。其余三个镜像也呈现同样的趋势，说明为了换取更快启动，离线构建成本与镜像体积增加都相当有限。冷启动结果上，论文报告 CoFS 在四个容器上都优于其他系统，而且背景预取往往会拖慢启动，因为它会和真正位于关键路径上的字节争抢网络与本地 I/O 资源。

更细的机制级实验也支撑了论文论点。相对于 `fuse-loopback`，CoFS 的平均查找延迟提升了 `73%` 到 `86%`；在 `elasticsearch` 容器上，完整路径并行查找又比关闭该优化的 CoFS 额外带来 `28%` 改善。对于缓存命中的读路径，作者构造了一个 `100 GB` 文件，先让它在容器里被完整读一次以确保下载完成，再清空 page cache 后运行 `fio`。结果显示 CoFS 与 `traditional`、`Nydus-erofs` 的顺序带宽和随机读延迟几乎一致，而明显优于 `Nydus-fuse` 和 `eStargz`，原因就在于缓存数据访问都留在了内核态。最后，`MPHF` 构建时间也不夸张：对一百万节点的随机图，平均耗时 `34.042 s`，最大值 `63.24 s`。考虑到大镜像本来就要花数分钟构建，这个离线代价是可以接受的。整体上，这组实验确实支持了 CoFS 在只读容器冷启动场景下的核心主张，但工作负载和硬件覆盖面仍然偏窄。

## 创新性与影响

相较 `Nydus` 和 `eStargz`，CoFS 的创新点不在于再发明一种 lazy-pull 格式，而在于针对“静态只读镜像树”这一前提重写元数据和缓存数据的热路径。它把构建期 `MPHF` 和运行期内核直读组合起来，使容器镜像文件系统第一次真正把“不可变”转化为查找性能收益。

因此，这篇论文会对容器 runtime、snapshotter、以及关注 serverless 与自动扩缩容冷启动的人有直接参考价值。它首先是一种新机制，其次也是一种清晰的设计观点：只要镜像是不可变的，就应该用离线工作换取在线关键路径的确定性与低开销。

## 局限性

CoFS 的适用范围高度依赖只读假设。整个 `MPHF` 布局都建立在文件树固定不变的前提上，因此它并不是通用文件系统方案，也不能直接处理可写层、动态变更镜像内容或需要在线重组的数据。即便如此，缓存未命中时它仍然要回退到用户态 daemon 与远端拉取，所以首次触碰某段数据时的长尾延迟依然受网络条件和 registry 行为影响。

实验覆盖面也比较有限。论文所有结果都来自单一硬件环境，网络是 `1 GbE`，底层介质是 HDD，冷启动工作负载也只有四个服务镜像。这足以证明机制有效，但不足以说明它在 SSD、更快网络、更大规模集群或更对抗性的访问模式下会如何表现。此外，CoFS 的工程复杂度不低：它要求自定义镜像元数据、扩展内核侧 `FUSE` 驱动、维护稀疏 mirror file，并依赖 `kprobe` 做路径优化，这些都可能提高长期维护成本。

## 相关工作

- _Harter et al. (FAST '16)_ — `Slacker` 说明了 lazy distribution 对容器启动延迟的重要性；CoFS 则进一步改造文件系统查找与缓存命中读路径来缩短同一条关键路径。
- _Li et al. (USENIX ATC '20)_ — `DADI` 采用块级懒加载容器镜像，而 CoFS 保持文件系统粒度，从而继续利用页缓存共享并按文件组织优化。
- _Bijlani and Ramachandran (USENIX ATC '19)_ — `ExtFUSE` 用内核内 eBPF 片段加速 `FUSE`，而 CoFS 借助离线 `MPHF` 预计算，把不可变镜像的大部分元数据查找直接从通用用户态路径里拿掉。
- _Cho et al. (FAST '24)_ — `RFUSE` 改善的是 `FUSE` 的内核-用户态通信底座，而 CoFS 专门针对容器镜像定制查找与缓存读路径，二者在思路上互补。
